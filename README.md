# CAN Sniffer Parser

Универсальный консольный инструмент на Python для пассивного чтения CAN-шины через разные CAN-адаптеры, автоматического определения доступных параметров соединения и сохранения трафика в отдельную папку сессии.

Ключевое требование проекта: адаптер всегда работает только в режиме `silent` / `listen-only`. Программа не должна подтверждать кадры ACK, передавать CAN-кадры, выполнять активное сканирование или каким-либо другим способом влиять на шину.

> Проект предназначен для диагностики и анализа оборудования, к которому у пользователя есть разрешённый доступ. Подключение должно выполняться с соблюдением требований электрической безопасности конкретной CAN-сети.

## Что будет делать программа

При запуске пользователь указывает тип сниффера и, при необходимости, известные параметры CAN-сети. После этого программа:

1. создаёт отдельную папку сессии;
2. открывает выбранный CAN-интерфейс строго в `listen-only` режиме;
3. определяет неизвестные параметры из заданного набора допустимых вариантов;
4. проверяет качество приёма без передачи данных в шину;
5. непрерывно записывает исходные CAN-кадры;
6. пытается определить используемые протоколы верхнего уровня;
7. декодирует известные сообщения по DBC и подключаемым декодерам;
8. сохраняет метаданные, диагностические события и статистику сессии;
9. корректно закрывает файлы при остановке по `Ctrl+C`, сигналу ОС или заданному тайм-ауту.

## Основные возможности

- единый CLI для нескольких типов CAN-снифферов;
- расширяемая система драйверов адаптеров;
- пассивный подбор скорости CAN;
- поддержка Classical CAN (CAN 2.0A/2.0B) и, если позволяет адаптер, CAN FD;
- распознавание стандартных и расширенных идентификаторов;
- запись кадров с временными метками;
- сохранение raw-логов без потери исходных данных;
- декодирование по DBC;
- подключаемые декодеры транспортных и прикладных протоколов;
- человекочитаемый вывод в JSON Lines и CSV;
- отдельная директория для каждой сессии;
- отсутствие любых операций передачи.

## Поддерживаемые адаптеры

Архитектура будет построена поверх абстракции драйвера. Первый набор реализаций планируется для интерфейсов, поддерживаемых библиотекой `python-can`, например:

- Zubax Babel через SLCAN;
- RaccoonLab SNIFFER с SLCAN-прошивкой через COM-порт;
- CANable/CandleLight на STM32F072 через SLCAN или GS-USB;
- SocketCAN;
- PEAK PCAN;
- Vector;
- Kvaser;
- IXXAT;
- CANable/candleLight через SocketCAN или совместимый backend;
- SLCAN-совместимые устройства.

Фактическая доступность зависит от ОС, драйвера производителя и способности конкретного устройства аппаратно работать в `listen-only`. Backend, для которого невозможно гарантированно включить silent-режим, программа должна отклонять, а не запускать в обычном режиме.

### Работа в Windows

Windows является основной целевой системой проекта. В Windows предполагается использовать нативные драйверы производителя или последовательный COM-порт:

- Zubax Babel — как SLCAN-устройство через виртуальный COM-порт;
- RaccoonLab SNIFFER — как SLCAN-устройство через виртуальный COM-порт;
- STM32F072 с CandleLight — напрямую через GS-USB/WinUSB;
- PEAK PCAN — через установленный PCAN-Basic;
- Vector — через установленный XL Driver Library;
- Kvaser — через установленный CANlib;
- IXXAT — через VCI-драйвер;
- SLCAN-совместимые устройства — через `COM1`, `COM2` и другие COM-порты.

SocketCAN относится к Linux и в Windows не используется. Перед первым запуском пользователь устанавливает драйвер своего адаптера, после чего может проверить обнаруженные интерфейсы командой `adapters list`. Программа должна выдавать понятное сообщение, если DLL производителя, драйвер или COM-порт не найден.

Новый адаптер добавляется как отдельный драйвер, реализующий общий интерфейс:

```text
probe()              проверить наличие устройства
capabilities()       получить CAN/CAN FD и silent-возможности
open(config)         открыть интерфейс только в listen-only
receive(timeout)     получить кадр
statistics()         получить счётчики ошибок и приёма
close()              закрыть интерфейс
```

## Режим silent / listen-only

Это обязательный и неотключаемый режим работы программы.

- В приложении не будет команды отправки кадров.
- Драйвер обязан явно подтвердить включение `listen-only`.
- Если адаптер или backend не может подтвердить silent-режим, запуск завершается ошибкой.
- Автоопределение параметров выполняется только прослушиванием и переключением локальной конфигурации контроллера.
- Программа не выполняет диагностические запросы, UDS-запросы, RTR-запросы или активное определение узлов.

Silent-режим приложения не заменяет корректное физическое подключение. Нельзя добавлять лишнюю терминацию или подключать адаптер с несовместимыми уровнями напряжения.

## Автоматический подбор параметров CAN

### Скорость Classical CAN

Если скорость не передана пользователем, программа последовательно проверяет заданный профиль скоростей, например:

```text
1 Mbit/s, 800, 500, 250, 125, 100, 83.333, 50, 33.333, 20, 10 kbit/s
```

Для каждой скорости интерфейс открывается в `listen-only`, после чего в течение короткого окна собираются:

- количество корректных кадров;
- количество ошибок контроллера, если backend предоставляет их;
- доля кадров с допустимым DLC;
- стабильность идентификаторов и интервалов;
- наличие трафика в нескольких последовательных окнах.

Выбирается конфигурация с устойчивым потоком валидных кадров и лучшей оценкой качества. Пользователь сможет ограничить набор кандидатов или сразу передать известную скорость.

```powershell
python -m can_sniffer capture --adapter socketcan --channel can0 --bitrate auto
python -m can_sniffer capture --adapter pcan --channel PCAN_USBBUS1 --bitrate 500000
python -m can_sniffer capture --adapter slcan --channel COM5 --bitrates 500000,250000,125000
```

Важно: при полностью молчащей шине скорость определить невозможно. Некоторые адаптеры также не предоставляют достаточно счётчиков ошибок для надёжного выбора. В таком случае программа сохранит результаты проверки и завершится с сообщением о неопределённой конфигурации либо потребует явный `--bitrate`.

### CAN FD

Для CAN FD отдельно задаются или подбираются nominal bitrate и data bitrate. Автоподбор возможен только для адаптеров, которые поддерживают CAN FD и безопасное переключение конфигурации в silent-режиме.

```powershell
python -m can_sniffer capture --adapter vector --channel 0 --fd auto
python -m can_sniffer capture --adapter vector --channel 0 --bitrate 500000 --data-bitrate 2000000
```

### Формат кадров

Standard ID (11 бит) и Extended ID (29 бит) не требуют активного подбора: формат определяется отдельно для каждого принятого кадра по признаку IDE. Аналогично фиксируются RTR, error frame, FD, BRS и ESI, если они доступны через выбранный backend.

## Определение протокола

CAN 2.0 и CAN FD определяют формат кадров, но не смысл байтов payload. Программа будет использовать пассивные эвристики и подключаемые анализаторы для обнаружения протоколов верхнего уровня, например:

- ISO-TP (ISO 15765-2);
- J1939;
- CANopen;
- NMEA 2000;
- собственные протоколы, описанные плагином или DBC.

Результат распознавания всегда содержит уровень уверенности и признаки, на которых он основан. Один поток может одновременно соответствовать нескольким гипотезам. Программа не должна выдавать эвристическое предположение за достоверно установленный протокол.

Без DBC, спецификации производителя или специализированного декодера невозможно универсально определить физический смысл произвольных байтов — например, отличить температуру от скорости или узнать коэффициент масштабирования. Поэтому предусмотрены два уровня представления:

1. структурное декодирование: ID, DLC, флаги, байты, целые значения, изменения битов, ISO-TP/J1939/CANopen-поля;
2. семантическое декодирование: имена сообщений и сигналов, единицы измерения, коэффициенты и диапазоны из DBC или плагина.

## Планируемый CLI

Основная команда:

```powershell
python -m can_sniffer capture [параметры]
```

Для быстрой проверки подключения предусмотрена максимально простая команда `dump`. По умолчанию она ждёт один CAN-кадр, печатает его в консоль и сразу завершает работу:

```powershell
python -m can_sniffer dump --adapter pcan --channel PCAN_USBBUS1 --bitrate 500000
```

Пример вывода:

```text
2026-06-22T14:35:43.123456+03:00  ID=0x18FEF100  EXT  DLC=8  DATA=FF 7D 32 10 FF FF FF FF
```

Команда `dump`, как и длительный захват, открывает адаптер только в `listen-only`. Она ничего не передаёт, не подтверждает кадр ACK и по умолчанию не создаёт папку с логами.

Если скорость неизвестна, разрешён пассивный автоподбор:

```powershell
python -m can_sniffer dump --adapter pcan --channel PCAN_USBBUS1 --bitrate auto
```

В этом случае до вывода кадра программа должна принять несколько кадров на этапе определения скорости. После выбора конфигурации она выводит первый следующий корректный кадр и завершается. Получить ровно один физический кадр со всей шины и одновременно достоверно определить неизвестную скорость невозможно.

Дополнительные варианты:

```powershell
# Получить пять кадров
python -m can_sniffer dump --adapter pcan --channel PCAN_USBBUS1 --bitrate 500000 --count 5

# Ждать кадр не более десяти секунд
python -m can_sniffer dump --adapter slcan --channel COM5 --bitrate 250000 --timeout 10

# Вывести один кадр в JSON
python -m can_sniffer dump --adapter vector --channel 0 --bitrate 500000 --format json
```

Для `dump` используются параметры `--adapter`, `--channel`, `--device`, `--bitrate`, `--bitrates`, `--fd` и `--data-bitrate`, общие с `capture`, а также:

| Параметр | Значение по умолчанию | Назначение |
|---|---:|---|
| `--count` | `1` | Количество кадров для вывода |
| `--timeout` | `30` | Максимальное время ожидания каждого кадра в секундах |
| `--tty-baudrate` | зависит от устройства | Скорость UART для SLCAN |
| `--autodetect-window` | `0.5` | Окно проверки кандидата при `--bitrate auto` |
| `--btr` | — | Пользовательские BTR0/BTR1 вместо стандартного bitrate |

При истечении `--timeout` без принятого кадра команда завершается с ненулевым exit code. Ошибки конфигурации и невозможность подтвердить silent-режим также приводят к ненулевому exit code.

Примеры:

```powershell
# Минимальный запуск с автоподбором скорости
python -m can_sniffer capture --adapter pcan --channel PCAN_USBBUS1

# Известная скорость и DBC-файл
python -m can_sniffer capture --adapter socketcan --channel can0 --bitrate 500000 --dbc vehicle.dbc

# Ограниченный список скоростей и описание сессии
python -m can_sniffer capture --adapter slcan --channel COM5 --bitrates 500000,250000,125000 --description "bench test"

# Остановка через 10 минут
python -m can_sniffer capture --adapter vector --channel 0 --bitrate 500000 --duration 10m
```

Планируемые параметры:

| Параметр | Назначение |
|---|---|
| `--adapter` | Backend/тип CAN-адаптера |
| `--channel` | Канал, интерфейс или COM-порт |
| `--device` | Серийный номер или идентификатор устройства |
| `--bitrate` | Скорость nominal phase или `auto` |
| `--bitrates` | Разрешённый список скоростей для автоподбора |
| `--fd` | Включить или автоматически определить CAN FD |
| `--data-bitrate` | Скорость data phase CAN FD |
| `--dbc` | Один или несколько DBC-файлов |
| `--decoder` | Дополнительный декодер протокола |
| `--output` | Корневой каталог сессий; по умолчанию `sessions` |
| `--description` | Обязательное краткое описание сессии для имени папки |
| `--duration` | Максимальная длительность записи |
| `--autodetect-window` | Время проверки одного кандидата скорости |
| `--log-level` | Уровень служебного журнала |

Также планируются команды:

```powershell
python -m can_sniffer adapters list
python -m can_sniffer adapters probe --adapter pcan
python -m can_sniffer inspect path\to\session
python -m can_sniffer decode path\to\raw.jsonl --dbc vehicle.dbc
```

Команда `decode` позволит повторно декодировать ранее записанные raw-данные без подключения к CAN-шине.

## Структура сессии

По умолчанию каждая сессия создаётся в каталоге `sessions`. Имя вложенной папки состоит из текущей локальной даты и описания, переданного через обязательный аргумент `--description`:

```text
sessions/
└── 2026-06-22_bench-test/
    ├── session.json
    ├── raw.jsonl
    ├── raw.asc
    ├── decoded.jsonl
    ├── decoded.csv
    ├── events.jsonl
    └── summary.json
```

Пример запуска:

```powershell
python -m can_sniffer capture --adapter pcan --channel PCAN_USBBUS1 --bitrate 500000 --description "bench test"
```

Описание нормализуется для безопасного имени папки Windows: пробелы заменяются дефисами, а символы `\ / : * ? " < > |` удаляются. Исходный текст без изменений сохраняется в `session.json`.

Если папка с такой датой и описанием уже существует, программа не перезаписывает её. Она создаёт следующую свободную папку с числовым суффиксом:

```text
sessions/
├── 2026-06-22_bench-test/
├── 2026-06-22_bench-test-2/
└── 2026-06-22_bench-test-3/
```

- `session.json` — параметры запуска, сведения об адаптере, выбранная скорость, версии программы и декодеров;
- `raw.jsonl` — канонический lossless-лог принятых кадров;
- `raw.asc` — совместимый текстовый лог для внешних CAN-инструментов;
- `decoded.jsonl` — структурированные декодированные сообщения;
- `decoded.csv` — плоское человекочитаемое представление сигналов;
- `events.jsonl` — смена состояния, ошибки, переподключения и результаты autodetect;
- `summary.json` — итоговая статистика сессии.

Запись выполняется потоково. Raw-кадр сохраняется до декодирования, поэтому ошибка декодера не приводит к потере исходного трафика.

## Формат сырых данных

Канонический формат `raw.jsonl` содержит одну JSON-запись на кадр:

```json
{"timestamp":"2026-06-22T14:35:43.123456789+03:00","monotonic_ns":8245123123456,"channel":"PCAN_USBBUS1","id":"0x18FEF100","extended":true,"fd":false,"brs":false,"esi":false,"rtr":false,"error":false,"dlc":8,"data":"FF7D3210FFFFFFFF"}
```

Для точного анализа сохраняются как системная временная метка, так и монотонное время. Если устройство предоставляет аппаратный timestamp, он также записывается отдельным полем.

## Формат декодированных данных

`decoded.jsonl` остаётся пригодным для машинной обработки, но содержит понятные имена и значения:

```json
{"timestamp":"2026-06-22T14:35:43.123456789+03:00","id":"0x18FEF100","protocol":"j1939","message":"EngineTemperature1","signals":{"EngineCoolantTemperature":{"value":85.0,"unit":"degC"}},"decoder":"vehicle.dbc","confidence":1.0}
```

Если семантика неизвестна, запись всё равно содержит структурное представление:

```json
{"timestamp":"2026-06-22T14:35:43.123456789+03:00","id":"0x123","protocol":"unknown","data_hex":"102A00FF","data_bytes":[16,42,0,255],"unsigned_le":4278200848,"changed_bits":[9,11],"confidence":0.0}
```

CSV предназначен для просмотра в табличных редакторах и содержит по строке на декодированный сигнал: timestamp, CAN ID, протокол, сообщение, сигнал, значение и единицу измерения.

## Предлагаемая архитектура

```text
CLI
 ├── конфигурация и проверка параметров
 ├── менеджер сессии
 └── pipeline захвата
      ├── adapter driver (только listen-only)
      ├── autodetect
      ├── raw writer
      ├── protocol detectors
      ├── DBC/plugin decoders
      └── decoded writers
```

Запись и декодирование будут разделены очередями с ограниченным размером. При высокой нагрузке приоритет имеет raw-лог. Переполнение очереди, пропуски кадров и ошибки backend обязательно отражаются в `events.jsonl` и `summary.json`.

Ожидаемые модули:

```text
src/can_sniffer/
├── cli.py
├── config.py
├── models.py
├── session.py
├── adapters/
├── autodetect/
├── capture/
├── protocols/
├── decoders/
└── writers/
```

## Надёжность и ограничения

- Абсолютно универсального декодирования неизвестного CAN payload не существует; для точных имён и физических величин нужен DBC или спецификация протокола.
- Пассивный autodetect требует наличия трафика на шине.
- Не все адаптеры позволяют получить счётчики ошибок в silent-режиме, поэтому качество автоподбора зависит от оборудования.
- Для нестандартных bit timing одного значения bitrate может быть недостаточно. В будущем конфигурация сможет принимать sample point, SJW и параметры сегментов.
- При высокой загрузке шины USB/serial-адаптер может терять кадры. Такие потери должны быть видимы в статистике, если устройство или драйвер предоставляет счётчик.
- Формат ASC может не сохранять все vendor-specific метаданные, поэтому `raw.jsonl` считается основным источником данных.
- Переключение локальной скорости в аппаратном silent-режиме не передаёт кадры, но конкретный backend всё равно должен пройти проверку возможностей перед использованием.

## Критерии готовности первой версии

Первая рабочая версия должна:

- поддерживать Windows как основную платформу;
- поддерживать минимум два разных backend;
- отказываться от запуска без подтверждённого `listen-only`;
- выводить один кадр и завершаться через команду `dump`;
- принимать известную скорость или пассивно выбирать её из списка;
- создавать уникальную папку каждой сессии;
- записывать все полученные кадры в `raw.jsonl` и ASC;
- декодировать сообщения по DBC;
- формировать `decoded.jsonl`, CSV и итоговую статистику;
- корректно завершать файлы при штатной остановке;
- иметь тесты, подтверждающие отсутствие вызовов передачи в capture pipeline.

## Текущий статус

В репозитории есть первый минимальный прототип `can_dump.py` для Windows. Он принимает известную скорость либо пассивно выбирает её из списка, открывает адаптер в аппаратном listen-only режиме, получает заданное число кадров и выводит их в консоль. Запись сессий и декодирование пока не реализованы.

Текущая совместимость:

| `--device` | Устройство/прошивка | Подключение | Silent-механизм |
|---|---|---|---|
| `zubax` | Zubax Babel | виртуальный `COM` | SLCAN-команда `L` с проверкой ACK |
| `raccoonlab` | RaccoonLab SNIFFER с прошивкой `can-uart`/SLCAN | виртуальный `COM` | SLCAN-команда `L` с проверкой ACK |
| `slcan` | STM32F072 с CANable/SLCAN или другой совместимый адаптер | виртуальный `COM` | SLCAN-команда `L` с проверкой ACK |
| `candlelight` | STM32F072 с CandleLight/GS-USB | USB/WinUSB | USB-флаг `GS_CAN_MODE_LISTEN_ONLY` с проверкой capability |
| `canable` / `gs_usb` | Алиасы для CANable с GS-USB/CandleLight | USB/WinUSB | USB-флаг `GS_CAN_MODE_LISTEN_ONLY` с проверкой capability |

Один и тот же STM32F072 может требовать `slcan` или `candlelight` в зависимости от залитой прошивки. Название платы этого не определяет.

Установка:

```powershell
python -m pip install -r requirements.txt
```

Пример для Zubax Babel:

```powershell
python .\can_dump.py --device zubax --channel COM5 --bitrate 500000
```

Пример для RaccoonLab:

```powershell
python .\can_dump.py --device raccoonlab --channel COM7 --bitrate 1000000
```

Пассивный автоподбор стандартной скорости:

```powershell
python .\can_dump.py --device raccoonlab --channel COM7 --bitrate auto
```

Порядок и набор кандидатов можно задать явно:

```powershell
python .\can_dump.py --device raccoonlab --channel COM7 --bitrate auto --bitrates 500000,250000,1000000 --autodetect-window 1
```

Для SLCAN-устройств, которые реализуют полный протокол LAWICEL, нестандартный bit timing задаётся парой регистров BTR0/BTR1 из четырёх hex-символов:

```powershell
python .\can_dump.py --device slcan --channel COM5 --btr 031C
```

Проверенная прошивка RaccoonLab `V0009` отклоняет `sBTR0BTR1`, поэтому профиль `raccoonlab` поддерживает только стандартные `S0…S8`. Скрипт отклоняет `--btr` для этого профиля до открытия CAN-канала. Для другого совместимого адаптера используется профиль `slcan`.

Для каждого переключения SLCAN-драйвер следует документированному порядку LAWICEL: синхронизация очереди, запрос версии `V`, закрытие `C`, установка `S0…S8` либо `sBTR0BTR1`, затем открытие исключительно командой `L`. После каждой команды проверяется ответ `CR`/`BELL`. Обычная активная команда `O` не используется.

Пример для STM32F072 с прошивкой CANable/SLCAN:

```powershell
python .\can_dump.py --device slcan --channel COM7 --bitrate 500000
```

Пример для STM32F072 с прошивкой CandleLight/GS-USB:

```powershell
python .\can_dump.py --device candlelight --channel 0 --bitrate 500000
```

Если Windows показывает устройство как `canable gs_usb`, можно использовать соответствующий алиас:

```powershell
python .\can_dump.py --device canable --channel 0 --bitrate 1000000
```

Пассивный автоподбор скорости CANable/GS-USB:

```powershell
python .\can_dump.py --device canable --channel 0 --bitrate auto
```

При автоподборе каждый кандидат запускается с аппаратным флагом `GS_CAN_MODE_LISTEN_ONLY`. Error frames не считаются подтверждением скорости; требуется поток обычных принятых CAN-кадров.

Для CandleLight `--channel` — индекс среди найденных GS-USB устройств, начиная с `0`. Для SLCAN это имя COM-порта из диспетчера устройств Windows.

Дополнительно можно указать `--count 5`, `--timeout 10` и скорость последовательного порта через `--tty-baudrate`. Для профиля `raccoonlab` по умолчанию используется UART 1 Мбит/с, для остальных SLCAN-устройств — 115200 бод. Если устройство не подтверждает последовательность `C/Sx/L` либо GS-USB прошивка не заявляет и не включает `LISTEN_ONLY`, скрипт завершается ошибкой. Он не пытается продолжить в активном режиме.
